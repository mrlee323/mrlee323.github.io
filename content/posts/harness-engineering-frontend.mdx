## 하네스 엔지니어링이란?

**하네스(Harness)**는 원래 등반·산업 현장에서 쓰는 안전 장비입니다. 소프트웨어 공학에서 하네스 엔지니어링은 이 개념을 그대로 가져옵니다.

> 시스템이 예상 밖으로 움직이더라도 추락하지 않도록 — 제어 가능한 범위 안에서 안전하게 실험하고 배포하는 구조를 설계하는 것

단순히 테스트를 작성하는 것과 다릅니다. **언제든 되돌릴 수 있고, 일부에게만 노출하며, 이상을 즉시 감지할 수 있는 인프라 전체**를 가리킵니다.

## 왜 프론트엔드에도 하네스가 필요한가?

프론트엔드는 오랫동안 "가장 바깥 레이어"로 취급받았습니다. 백엔드가 안정적이면 프론트는 그냥 보여주기만 한다는 인식이 있었죠.

하지만 현실은 다릅니다.

- 결제 UI 버그 하나가 매출 직격탄
- A/B 테스트를 위해 배포를 반복하는 비용
- SSR 환경에서 런타임 에러가 페이지 전체를 날려버리는 상황
- 사용자마다 다른 기기·네트워크 환경

프론트엔드야말로 하네스가 가장 절실한 영역입니다.

## 하네스 엔지니어링의 4가지 축

### 1. 피처 플래그 (Feature Flags)

코드를 배포하는 것과 기능을 노출하는 것을 분리합니다.

```tsx
// hooks/useFeatureFlag.ts
const FLAGS: Record<string, boolean> = {
  NEW_CHECKOUT_FLOW: process.env.NEXT_PUBLIC_FLAG_NEW_CHECKOUT === 'true',
  AI_SEARCH: process.env.NEXT_PUBLIC_FLAG_AI_SEARCH === 'true',
};

export function useFeatureFlag(flag: string): boolean {
  return FLAGS[flag] ?? false;
}
```

```tsx
// 사용 예시
function CheckoutButton() {
  const isNewFlow = useFeatureFlag('NEW_CHECKOUT_FLOW');

  return isNewFlow ? <NewCheckoutButton /> : <LegacyCheckoutButton />;
}
```

피처 플래그의 핵심은 **코드가 배포된 상태에서도 스위치처럼 켜고 끌 수 있다**는 점입니다. 이상이 생기면 코드 롤백 없이 플래그만 내리면 됩니다.

**사용자 단위 제어**가 필요하다면 서버 기반 플래그(LaunchDarkly, Vercel Edge Config 등)로 확장할 수 있습니다.

```ts
// Vercel Edge Config 기반 예시
import { get } from '@vercel/edge-config';

export async function getFeatureFlags(userId: string) {
  const rollout = await get<number>('new_checkout_rollout'); // 0~100
  const hash = hashUserId(userId) % 100;
  return { NEW_CHECKOUT_FLOW: hash < (rollout ?? 0) };
}
```

### 2. 에러 바운더리 (Error Boundary)

React의 에러 바운더리는 가장 기본적인 하네스입니다. 컴포넌트 트리 일부가 폭발해도 전체 페이지가 죽지 않도록 격리합니다.

```tsx
// components/ErrorBoundary.tsx
'use client';

import { Component, type ReactNode } from 'react';

interface Props {
  fallback?: ReactNode;
  onError?: (error: Error) => void;
  children: ReactNode;
}

interface State {
  hasError: boolean;
  error: Error | null;
}

export class ErrorBoundary extends Component<Props, State> {
  state: State = { hasError: false, error: null };

  static getDerivedStateFromError(error: Error): State {
    return { hasError: true, error };
  }

  componentDidCatch(error: Error) {
    this.props.onError?.(error);
    // Sentry, Datadog 등으로 전송
    reportError(error);
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback ?? <DefaultErrorUI error={this.state.error} />;
    }
    return this.props.children;
  }
}
```

```tsx
// 레이아웃 단위로 감싸기
<ErrorBoundary fallback={<RecommendSection />}>
  <PersonalizedHero userId={userId} />
</ErrorBoundary>
```

중요한 것은 **경계의 단위를 어디에 두느냐**입니다. 너무 넓으면 복구 범위가 크고, 너무 좁으면 UX가 파편화됩니다. 비즈니스 임팩트 기준으로 나누는 것이 좋습니다.

### 3. 테스트 하네스 (Test Harness)

테스트 하네스는 **실제 환경을 흉내 낸 제어된 실험 환경**입니다. 단순히 유닛 테스트를 쓰는 것을 넘어, 외부 의존성을 교체 가능하게 설계하는 것이 핵심입니다.

#### MSW로 API 하네스 구성

```ts
// mocks/handlers.ts
import { http, HttpResponse } from 'msw';

export const handlers = [
  http.get('/api/checkout', () => {
    return HttpResponse.json({ status: 'success', orderId: 'test-123' });
  }),

  // 실패 시나리오도 하네스로 제어
  http.post('/api/payment', () => {
    return HttpResponse.json({ error: 'CARD_DECLINED' }, { status: 402 });
  }),
];
```

```ts
// 특정 테스트에서만 오버라이드
server.use(
  http.post('/api/payment', () =>
    HttpResponse.json({ error: 'NETWORK_TIMEOUT' }, { status: 503 })
  )
);
```

#### Storybook을 하네스로

Storybook은 단순한 컴포넌트 문서 도구가 아닙니다. 각 Story가 곧 **재현 가능한 상태 하네스**입니다.

```tsx
// CheckoutButton.stories.tsx
export const PaymentPending: Story = {
  args: { status: 'pending' },
};

export const PaymentFailed: Story = {
  args: { status: 'failed', error: 'CARD_DECLINED' },
};

export const PaymentSuccess: Story = {
  args: { status: 'success' },
};
```

QA, 디자이너, 개발자가 모두 같은 환경에서 엣지 케이스를 재현할 수 있습니다.

### 4. 카나리 배포 & 점진적 롤아웃

전체 사용자에게 한 번에 배포하지 않고, 일부에게 먼저 노출해 이상이 없으면 점진적으로 확장합니다.

```ts
// middleware.ts (Next.js)
import { NextRequest, NextResponse } from 'next/server';

export function middleware(req: NextRequest) {
  const bucket = getBucket(req.cookies.get('uid')?.value ?? '');

  // 10% 사용자에게 새 페이지 노출
  if (bucket < 10) {
    return NextResponse.rewrite(new URL('/v2/checkout', req.url));
  }

  return NextResponse.next();
}

function getBucket(uid: string): number {
  let hash = 0;
  for (const char of uid) hash = (hash * 31 + char.charCodeAt(0)) % 100;
  return hash;
}
```

Vercel을 쓴다면 **Skew Protection**과 **Edge Config**를 조합해 배포 중 요청이 엇갈리는 문제도 방어할 수 있습니다.

## 프론트엔드 하네스 설계 체크리스트

| 레이어 | 도구 / 방법 | 목적 |
|--------|------------|------|
| 코드 격리 | Error Boundary | 런타임 폭발 범위 제한 |
| 기능 제어 | Feature Flag | 코드 배포와 기능 노출 분리 |
| 의존성 교체 | MSW, 의존성 주입 | 외부 API 없이 재현 가능한 환경 |
| 시각적 격리 | Storybook | 컴포넌트 단위 상태 하네스 |
| 배포 제어 | 카나리, Edge Middleware | 트래픽 단위 점진적 롤아웃 |
| 이상 감지 | Sentry, Web Vitals | 하네스가 작동했을 때 알림 |

## 정리

하네스 엔지니어링의 핵심 철학은 간단합니다.

**"언제든 되돌릴 수 있는가?"**

배포 전 테스트만으로는 부족합니다. 프로덕션에서 이상이 생겼을 때 플래그를 내리고, 에러 경계가 격리하고, 모니터링이 즉시 알려주는 구조가 함께 있어야 진짜 안전망입니다.

프론트엔드도 이제 인프라입니다. 화면을 만드는 것을 넘어 **제어 가능한 시스템을 설계하는 것** — 그게 하네스 엔지니어링이 프론트엔드 개발자에게 주는 관점의 전환입니다.
